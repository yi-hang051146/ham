# 项目开发经验总结

## 核心经验教训

### 1. 数据格式理解的重要性

**问题**：最初误以为思源笔记存储的是 HTML 格式，导致创建了不必要的 HTML 转换逻辑。

**真相**：思源笔记是**原生 Markdown** 格式（`.sy` 文件）。

**教训**：
- ✅ 在处理第三方工具数据前，**务必先确认数据格式**
- ✅ 使用官方 API 文档或直接测试验证数据格式
- ✅ 思源笔记通过 MCP 获取数据时，使用 `mode='markdown'` 参数

**最佳实践**：
```javascript
// ✅ 正确：获取 Markdown 格式
mcp__document(action='get_doc', id='xxx', mode='markdown')

// ❌ 错误：假设是 HTML 格式
// 不要使用返回 HTML DOM 的 API
```

---

### 2. API vs MCP 的使用场景

**API（应用程序接口）**：
- 使用场景：在 Vue 应用代码中
- 调用方式：HTTP 请求
- 适用对象：前端应用、后端服务

**MCP（Model Context Protocol）**：
- 使用场景：在 AI 助手中
- 调用方式：工具函数调用
- 适用对象：AI 助手、自动化脚本

**最佳实践**：
- 数据处理和自动化：使用 MCP（通过 AI 助手）
- 用户访问和实时交互：使用 API（在 Vue 应用中）

---

### 3. 静态数据 vs 动态数据

**静态数据**（当前方案）：
- 优点：适合 GitHub Pages 部署，无需后端
- 缺点：需要手动更新数据文件
- 适用场景：个人主页、静态网站

**动态数据**（通过 API/MCP）：
- 优点：数据实时更新
- 缺点：需要后端服务或 API 访问
- 适用场景：实时应用、协作系统

**最佳实践**：
- 对于 GitHub Pages 部署，使用静态数据
- 通过 MCP 定期更新静态数据文件
- 保持数据文件为 Markdown 格式，便于渲染

---

### 4. 文件组织原则

**模块化设计**：
```
src/
├── components/     # 按功能分类
├── services/       # API 服务层
├── utils/          # 工具函数
└── data/           # 静态数据
```

**清理原则**：
- ✅ 定期检查未使用的文件
- ✅ 删除空目录和示例文件
- ✅ 保持项目结构简洁

**命名规范**：
- 组件：PascalCase（`AppHeader.vue`）
- 服务：camelCase（`siyuanService.js`）
- 工具：camelCase（`formatDate.js`）

---

### 5. 工具链选择

**文件转换**：
- markitdown：各种格式 → Markdown
- 适用：PDF、Word、HTML 等导入思源笔记

**数据提取**：
- MCP：直接获取思源笔记 Markdown 内容
- 适用：数据更新、批量处理

**渲染**：
- marked：Markdown → HTML
- KaTeX：数学公式渲染

## 可复用的 Rules

### Rule 1: 数据格式验证
```
在处理第三方工具数据前，必须：
1. 查阅官方文档确认数据格式
2. 使用测试数据验证格式
3. 选择正确的 API 端点
```

### Rule 2: API 选择原则
```
选择 API 时：
- 需要用户实时访问 → 使用 HTTP API
- 需要 AI 助手处理 → 使用 MCP 工具
- 需要静态部署 → 预生成数据文件
```

### Rule 3: 思源笔记数据处理
```
处理思源笔记数据：
1. 数据格式：原生 Markdown（.sy 文件）
2. 获取方式：MCP + mode='markdown'
3. 渲染方式：marked + KaTeX
4. 不要使用 HTML DOM API
```

### Rule 4: 项目维护
```
定期维护项目：
1. 检查并删除未使用的文件
2. 更新文档以反映实际结构
3. 保持代码简洁，避免过度设计
4. 只保留必要的依赖
```

### Rule 5: 文档更新
```
当项目结构变化时：
1. 更新 README.md
2. 更新 PROJECT_STRUCTURE.md
3. 添加必要的说明和示例
4. 区分用户指南和开发文档
```

## 技术决策记录

### 决策 1：使用静态数据而非实时 API
- **原因**：需要部署到 GitHub Pages
- **结果**：数据文件更小，加载更快
- **权衡**：需要手动更新数据

### 决策 2：使用 MCP 而非 Python 脚本更新数据
- **原因**：MCP 更直接，无需认证配置
- **结果**：数据更新更简单
- **权衡**：依赖 AI 助手

### 决策 3：保留 Markdown 渲染器而非 HTML 转换
- **原因**：思源笔记原生 Markdown
- **结果**：代码更简洁，渲染效果更好
- **权衡**：无

---

## 避免的陷阱

1. ❌ **假设数据格式**：总是先验证
2. ❌ **过度设计**：只实现必要功能
3. ❌ **忽略文档**：保持文档同步更新
4. ❌ **保留冗余代码**：定期清理
5. ❌ **混淆 API 和 MCP**：明确使用场景

---

## 总结

核心原则：
1. **验证优先**：不确定时先验证
2. **保持简单**：避免过度设计
3. **文档同步**：代码和文档一起更新
4. **定期清理**：保持项目整洁
5. **明确场景**：选择合适的工具

这些经验可以应用到其他项目中，特别是涉及第三方工具集成和静态网站部署的场景。
